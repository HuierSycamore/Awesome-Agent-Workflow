### 代码风格-命名
✅ 命名统一采用驼峰风格：大小写字母混用，单词连在一起，不同单词间通过单词首字母大写来分开。
具体规则如下：

|类别|命名风格|形式举例|
|-|-|-|
| 函数名，类型名 | 大驼峰（可带模块前缀） | AaaBbb, MOD_AaaBbb（带模块前缀） |
| 变量，参数，宏参数，结构成员名 | 小驼峰 | aaaBbb |
| 全局变量名 | 带 'g_' 前缀小驼峰 | g_aaaBbb |
| 宏名，枚举名，goto 标签名 | 全大写，下划线分割 | AAA_BBB |
| 函数式宏名 | 全大写下划线分割，或大驼峰，或带模块前缀的大驼峰 | AAA_BBB, AaaBbb, MOD_AaaBbb（带模块前缀） |
| 常量名 | 全大写下划线分割，或带 'g_' 前缀的小驼峰 | AAA_BBB, g_aaaBbb |

- 对于全局函数、全局变量、宏、类型名、枚举名的命名，应当精确描述并全局唯一，避免冲突
- 对于函数局部变量，或者结构体、联合体中的成员变量，其命名在能够准确表达含义的前提下应该尽量简短

**关于"模块前缀"**：
- 仅大驼峰命名风格的符号（函数名，类型名），可选加模块名前缀。模块前缀尽量简短且不超过**2**级（下划线分隔模块前缀，如：MOD_SUBMOD_AaaBbb）。

### 代码风格-注释
✅ 使用流利的中文进行注释，在代码的功能、意图层次上进行注释，不要重复代码信息。
❌ 禁止写空有格式的函数头注释，不写显而易见对代码动作描述的注释。

### 代码风格-格式
✅ 每行字符数不超过 **120** 个
✅ 换行时将操作符留在行末，新行缩进一层或进行同类对齐
当语句过长，或者换行后有更好的可读性时，应根据层次或操作符优先级选择合适的断行点进行换行，并将表示未结束的操作符或连接符号留在行末。
函数参数列表举例：
```c
ReturnType result = FunctionName(paramName1, paramName2,
    paramName3, paramName4, paramName5);     // 符合：参数换行，4空格缩进（缩进对齐）
```

✅ 声明或定义指针变量或者返回指针类型函数时，"*"宜靠右跟随变量或者函数名。
**【正例】**
```c
int *p1;                     // 符合："*"跟随变量名
struct Foo *CreateFoo(void); // 符合: "*"跟随函数名
```
- 当"*"与变量或函数名之间有其他修饰符时，不宜紧跟修饰符。
```c
char * const VERSION = "V100";      // 符合: 当有 const 修饰符时，"*"两边都有空格
int Foo(const char * restrict p);   // 符合: 当有 restrict 修饰符时，"*"两边都有空格
```
**【例外】**
- 当右侧没有变量或函数名时，"*" 可以跟随类型
```c
sz = sizeof(int*); // 符合：右侧没有变量，"*"可以跟随类型
```

✅ 用空格突出关键字和重要信息，行末不加空格。
✅ 保持代码紧凑，避免过多空行，杜绝连续3个及以上空行。

### 编程实践-预处理
✅ 使用函数代替函数式宏，优先使用函数而非函数式宏。
✅ 定义宏时，要使用完备的括号。
✅ 包含多条语句的函数式宏的实现语句必须放在 do-while(0) 中。
**【正例】**：
```c
#define FOO(x) do { \
    (void)printf("arg is %d\n", (x)); \
    DoSomething((x)); \
} while (0)
```

❌ 禁止把带副作用的表达式作为参数传递给函数式宏。
✅ 函数式宏定义中慎用 return、goto、continue、break 等改变程序流程的语句。
❌ 禁止宏定义以分号结尾
❌ 禁止宏定义依赖宏外部的局部变量名
✅ `#if` 或 `#elif` 预处理指令中的常量表达式被求值前应确保其使用的标识符已定义，常量表达式的值应为布尔值

### 编程实践-头文件
✅ 在头文件中声明需要对外公开的接口。内部使用的函数声明，宏、枚举、结构体等定义不应放在头文件中。
❌ 禁止头文件循环依赖
❌ 禁止包含用不到的头文件
✅ 头文件应当自包含，就是头文件内涉及到的符号依赖其它头文件则需要把其它头文件包含到本头文件里，使得任意一个头文件均可独立编译。
✅ 头文件必须采取保护措施，防止重复包含
假定 VOS 工程的 timer 模块的 timer.h，其目录为 vos/include/timer.h。其保护符若使用 'TIME_H'很容易不唯一，所以使用项目源代码树的全路径，如：
```c
#ifndef VOS_INCLUDE_TIMER_H
#define VOS_INCLUDE_TIMER_H

...

#endif
```

❌ 禁止通过声明的方式引用外部函数接口、变量（测试代码例外）。
✅ 按照“C标准库，操作系统库，平台库，项目公共库，自己其他的依赖”的顺序包含头文件。

### 编程实践-数据类型
❌ 禁止重复定义基础类型，使用标准类型定义（如int32_t）
✅ 使用恰当的基本类型作为操作符的操作数（如数组下标采用整型，不用浮点型）
✅ 避免隐式有损算术转换
在必须做数据有损的算术转换时，应明确使用显式类型转换。
```c
double a = 1.5;
int i = (int)(a); // 符合
```

### 编程实践-常量
❌ 禁止使用小写字母"l"作为数值型常量后缀
✅ 不使用难以理解的字面量，magic number采用能表达业务含义的常量替代

### 编程实践-变量
❌ 禁止读取未经初始化的变量
✅ 避免大量栈分配。局部变量占用的栈空间过大时可能导致栈溢出错误，较大的临时缓存使用动态分配内存且在用完后释放。
✅ 慎重使用全局变量，如不可避免需要使用全局变量，需要注意全局变量避免作为模块接口使用。
✅ 指向资源句柄或描述符的变量（包括指针、文件描述符、socket描述符等），在资源释放后立即赋予新值，以避免发生使用已释放资源的安全问题。
✅ 避免将只在一个函数中使用的变量声明为全局变量，考虑用函数内静态变量替代。
✅ 资源（内存、文件、进程、线程、信号量、socket等）不再使用时应予以关闭或释放，异常路径也需要关注资源释放。

### 编程实践-表达式
✅ 执行算术运算或比较操作的两个操作数要求具有相同的基本类型，避免隐式转换副作用。
❌ 禁止在含有变量自增或自减运算的表达式中再次引用该变量
✅ 用括号明确表达式的操作顺序，避免过分依赖默认优先级

### 编程实践-控制语句
✅ 控制表达式的结果应是布尔值
✅ &&和||操作符的右侧操作数不应包含副作用，避免右侧表达式有时未被求值与初始意图不符。
✅ switch语句要有default分支
✅ switch语句避免依赖隐式下沉，需下沉的场景使用fall-through注释。

### 编程实践-整数
✅ 确保有符号整数运算不溢出
如下代码示例中，按照最大允许值进行校验，防止整数溢出。在编程时可根据具体业务场景做更严格的值域校验。
```c
int numA = ... // 来自外部数据
int numB = ... // 来自外部数据
int sum = 0;
if (((numA > 0) && (numB > (INT_MAX - numA))) ||
    ((numA < 0) && (numB < (INT_MIN - numA)))) {
    ... // 错误处理
}
sum = numA + numB;
...
```

✅ 确保无符号整数运算不回绕
**【正例】**
由于readLen变量记录的是已经处理报文的长度，必然会小于totalLen，因此将代码中的加法运算修改为减法运算，避免条件绕过。
```c
size_t totalLen = ... // 报文的总长度
size_t readLen = 0;   // 记录已经处理报文的长度
...
size_t pktLen = ParsePktLen(); // 来自网络报文
if (pktLen > totalLen - readLen) {
  ... // 错误处理
}
...
readLen += pktLen;
...
```

✅ 确保除法和余数运算不会导致除零错误(被零除)
✅ 整型表达式比较或赋值为一种更大类型之前必须用这种更大类型对它进行求值
**【正例】**
```c
int main(int argc, char *argv[])
{
    unsigned int a = 0x10000000;
    unsigned long long b = (unsigned long long)a * 0xab;  //符合，对a进行大类型求值，否则b值高位丢失
    printf("b = %llX\n", b);
    return 0;
}
```

✅ 校验外部数据中整数值的合法性
对源自外部数据的所有整数值，我们应执行严格评估，以确定它们是否具有可被识别的上限和下限。
对源自污染源的整数值限制其输入在合法范围内有助于防止整数溢出，回绕，截断和其他直接或间接由整数值限定不当引起的风险，如指针偏移越界、缓冲区溢出等。
✅ 移位操作符的右操作数必须是非负数且小于左操作数的基本类型位宽。
✅ 确保整数转换不会造成数据截断或符号错误。

### 编程实践-指针和数组
✅ 外部数据作为数组索引时必须确保在数组大小范围内。
当数据作为数组索引访问内存时，必须确保该数据始终在数组的有效范围内。尤其对于来自外部的数据，无论直接或间接作为数组索引，都要对其大小做严格校验。
❌ 禁止通过对数组类型的函数参数变量进行sizeof来获取数组大小。
❌ 禁止通过对指针变量进行sizeof操作来获取数组大小。
✅ 不同类型的对象指针之间不应进行强制转换（避免出现地址不对齐，数据长度不足等问题）。
✅ 声明一个带有外部链接的数组时，必须显式指定它的大小
❌ 禁止直接使用可能为空的指针，防止出现空指针解引用

### 编程实践-字符串
✅ 确保字符串存储有足够的空间容纳字符数据和null结束符，并确保字符串有null结束符。

### 编程实践-断言
✅ 避免在代码中直接使用assert()，需在宏里引用，宏仅在DEBUG版本打开。
❌ 禁止用断言检测程序在运行期间可能导致的错误（可能发生的错误要有错误处理代码）。

### 编程实践-函数设计
✅ 对所有外部数据进行合法性校验
外部数据（网络、用户输入、文件、API参数等）在使用前必须进行合法性校验，包括数据长度、范围、类型和格式校验。跨信任边界（跨模块/跨进程/跨环境）传递过来的数据必须校验。
对来自程序外部的数据要校验处理后才能使用。典型的使用场景包括：
- **作为数组索引**：将不可信的数据作为数组索引，可能导致超出数组上限，从而造成非法内存访问。
- **作为内存偏移地址**：将不可信数据作为指针偏移访问内存，可能造成非法内存访问，并可以造成进一步的危害，如任意地址读/写。
- **作为内存分配的尺寸参数**：例如进行0字节长度分配可能造成非法内存访问，或未限制分配内存大小造成的过度资源消耗。
- **作为循环条件**：将不可信数据作为循环限定条件，可能会引发缓冲区溢出、内存越界读/写、死循环等问题。
- **作为除数**：参见除零错误(被零除)。
- **作为命令行参数**：参见"禁止外部可控数据作为进程启动函数的参数"。
- **作为数据库查询语句的参数**：参见"禁止直接使用外部数据拼接SQL命令"。
- **作为输入/输出格式化字符串**：参见"调用格式化输入/输出函数时，禁止format参数受外部数据控制"。
- **作为内存拷贝长度**：当作为拷贝长度时，可能造成目标缓冲区溢出。
- **作为文件路径**：直接打开不可信路径，可能会导致目录遍历攻击，操作了攻击者无权操作的文件，使得系统被攻击者所控制。

✅ 函数内进入异常分支可能影响外部处理逻辑时需要有返回值告知调用者。
✅ 对象或函数的所有声明必须与定义具有一致的名称和类型限定符，如函数在.c中定义形式（参数名称&类型、返回值类型等）与.h中声明的形式保持一致。
✅ 函数的返回信息优先使用返回值而不是输出参数。
✅ 函数的指针参数如果不是用于修改所指向的对象就应该声明为指向const的指针
✅ 函数复杂度要求：代码行数不超过50行（非空非注释），参数个数不超过5个，最大代码嵌套深度不超过4层。
✅ 内联函数要尽可能短，避免超过10行
✅ 数组作为函数参数时，必须同时将其长度作为函数的参数
✅ 字符串或指针作为函数参数时，在函数体中应检查参数是否为NULL。如果函数仅用于处理模块内部的可信数据，需要由上层调用者保证该参数不能为NULL。
✅ 函数只在一个文件内使用时应当使用static修饰符

### 编程实践-函数使用
✅ 处理函数的返回值。
✅ 调用格式化输入/输出函数时，禁止format参数受外部数据控制，避免造成格式化字符串漏洞。
✅ 调用格式化输入/输出函数时，使用有效的格式字符串
在创建格式化字符串时的常见错误包括：
- format中参数个数与实参个数不一致；
- 使用无效的转换指示符；
- 使用与转换指示符不兼容的标志字符；
- 使用与转换指示符不兼容的长度修饰符；
- format中转换指示符与实参类型不匹配；
- 使用实参指定宽度或者精度时，实参的类型不是int类型；

❌ 禁止使用realloc()函数，使用malloc()加memcpy_s()组合来代替realloc()功能。
**【反例】**
```c
...
char *ptr = ...;    // ptr指向一段已经申请成功的内存
ptr = (char *)realloc(ptr, NEW_SIZE); // 当realloc()分配内存失败时会返回NULL，导致内存泄漏
if (ptr == NULL) {
    ... // 错误处理
}
```

**【正例】**
使用malloc()函数代替realloc()函数。
```c
// 使用malloc()函数代替realloc()函数
char *newPtr = (char *)malloc(NEW_SIZE);
if (newPtr == NULL) {
    ... // 错误处理
}

errno_t ret = memcpy_s(newPtr, NEW_SIZE, oldPtr, oldSize);
... // 校验ret，确保安全函数执行成功

... // 返回前，释放oldPtr
```
❌ 禁止使用alloca()函数申请栈上内存
✅ 必须检查安全函数返回值，并进行正确的处理（记录日志、返回错误等）
**【例外】**
（1）对固定长度的数组进行初始化，或对固定长度的结构体进行内存初始化
（2）函数参数中有指向内存的指针参数和对应的内存大小的参数时，对该内存进行初始化
（3）从堆中分配内存后，赋予初值
（4）根据源内存的大小进行同等大小的内存复制
（5）源缓冲区全部是静态字符/字符串常量且目的缓冲区总大小能完全容纳源缓冲区中的数据及null结束符
（6）在安全函数返回值检查错误处理代码中，如果又调用了安全函数，可以不进行返回值检查。

✅ 正确设置安全函数中的destMax参数
destMax有最大使用限制，如：`memset_s`、`memcpy_s`中最大限制为`SECUREC_MEM_MAX_LEN`，使用时禁止超过该最大限制。
关于安全函数，关于安全函数destMax参数，可参考securec.h，securectype.h
✅ 只能使用华为安全函数库中的安全函数（关于华为安全函数库，可参考securec.h）
❌ 禁止外部可控数据作为进程启动函数（system、popen、execl等）的参数。
❌ 禁止外部可控数据作为dlopen/LoadLibrary等模块加载函数的参数。
❌ 禁止直接使用外部数据拼接SQL命令
❌ 禁止在信号处理例程中调用非异步安全函数（信号处理程序中只调用POSIX标准声明的异步安全函数）
✅ 使用库函数时避免竞争条件
下表中左列的库函数在被多个线程调用时可能存在竞争条件问题。

| 函数 | 安全措施 |
| :-- | :-- |

### 编程实践-内存
✅ 内存申请前，必须对申请内存大小进行合法性校验
当申请内存大小由程序外部输入时，内存申请前，要求对申请内存大小进行合法性校验，防止申请0长度内存，或过多地、非法地申请内存。
大小不正确的参数、不当的范围检查、整数溢出或者截断都可能造成实际分配的缓冲区不符合预期。

✅ 内存分配后必须判断是否成功。
✅ 外部输入作为内存操作相关函数的复制长度时，需要校验其合法性。
✅ 内存中的敏感信息（口令、秘钥等）使用完毕后立即清0
❌ 禁止访问已释放的内存

### 编程实践-文件
✅ 创建文件时必须显式指定合适的文件访问权限
✅ 外部数据参与的文件路径使用前必须进行规范化并校验。
✅ 不要在共享目录中创建临时文件
如程序在Linux系统的共享目录/tmp下创建临时文件来保存临时数据，且文件名是硬编码的。由于文件名是硬编码的，因此是可预测的，攻击者只需用符号链接替换文件，然后链接所引用的目标文件就会被打开并写入新内容。

### 编程实践-并发与并行
✅ 避免数据竞争
数据竞争的发生通常是线程之间有耦合，如：一个线程读取或修改一个资源（比如：共享内存、全局变量等），同时，另一个线程在修改这个资源。在开发多线程/进程过程中，如果涉及对同一资源的操作，必须考虑数据竞争问题。

✅ 避免死锁
死锁的一种常见原因是多个线程之间互相持有彼此等待的锁而又不释放自己所持有的锁。
死锁导致程序卡死，需要在编码时避免死锁发生。

### 编程实践-其它
✅ 删除无效或永不执行的代码
✅ 不要在信号处理函数中访问共享对象
如果在信号处理程序中访问和修改共享对象，可能会造成竞争条件，使数据处于不确定的状态。
这条规则有两个不适用的场景：
- 读写无锁原子对象（Lock-Free Atomic Objects）;
- 读写volatile sig_atomic_t类型的对象，因为具有volatile  sig_atomic_t类型的对象即使在出现异步中断的时候也可以作为一个原子实体访问，是异步安全的。

❌ 禁止使用rand函数产生用于安全用途的伪随机数
C语言标准库rand()函数生成的是伪随机数，所以不能保证其产生的随机数序列质量。
典型的安全用途场景包括(但不限于)以下几种：
- 会话标识SessionID的生成；
- 挑战算法中的随机数生成；
- 验证码的随机数生成；
- 用于密码算法用途（例如用于生成IV、盐值、密钥等）的随机数生成。

❌ 禁止在发布版本中输出对象或函数的地址，如：将变量或函数的地址输出到客户端、日志、串口中。
❌ 禁止代码中包含公网地址（ip地址、域名等）。
❌ 禁止在用户界面、日志中暴露不必要信息（软件信息、会话标识、个人信息、口令/私钥等）。

    ...
}
